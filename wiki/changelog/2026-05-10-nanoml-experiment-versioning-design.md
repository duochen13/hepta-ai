# Design: NanoML — Experiment-Level Data Versioning for ML Researchers

## Problem Statement

ML researchers iterate on preprocessing strategies (deduplication, filtering, sampling rates) to improve model metrics, but lack tooling to track which data variant produced which model result. When running experiment sweeps (trying v0=raw, v1=deduped, v2=negative_sampled), they manually track relationships between data versions and model runs, making it impossible to answer "which preprocessing improved NE the most?" without manual correlation.

**Concrete example**: Researcher trains recommendation models on three preprocessing variants:
- v0: raw user-item interactions
- v1: deduplicated interactions
- v2: negative sampling at rate 0.8

Each variant trains 4 models with different hyperparameters. Without NanoML, there's no single view showing which (data variant, hyperparameter) combination produced the best NE/CTR. MLflow tracks runs, but doesn't version the input data.

## Demand Evidence

**Observed at current/former employer**: ML teams manually debug production regressions (NE drops for like prediction). When NE drops, engineers check if other models' NE dropped, then backtrace through CTR shift, coverage shift, etc. **Time wasted: finding which upstream data shifted and root cause took days**.

**Specific incident**: Two researchers independently cleaned the same dataset, created merge conflicts. Senior ML engineer had to step in and reconcile.

**Value quantification**: Would save 50%+ of research time — easily worth $50k-200k/year for an ML team.

**Failure modes observed with agent-driven research**:
1. Context/memory loss across experiment iterations
2. Bad reward signals (agent optimizes wrong metric)
3. Reproducibility collapse (can't recreate "good" run from 3 days ago)
4. Compute runaway (agent spawns unbounded experiments)
5. Human-agent collaboration gap (can't interpret agent's exploration)

## Status Quo

**Current tools**:
- **MLflow**: Tracks experiment runs (metrics, params, models) but NOT input datasets
- **Weights & Biases**: Similar to MLflow, run-centric tracking
- **Manual dataset tracking**: Researchers save CSVs with version numbers in filenames (`data_v1.csv`, `data_v2.csv`), manually correlate with MLflow runs via tags

**What they're doing manually**:
- Separate dashboards for each layer — Datadog for metrics, SQL for data distributions, manual correlation
- Mix of roles debugging — on-call starts investigation, escalates to ML engineer, platform team helps with data queries
- Understanding WHY data shifted (finding root cause upstream) took days

**Surprise from observation**: "They spent way more time on understanding WHY the data shifted than I expected" — the root cause analysis, not just detecting the shift, consumed days.

## Target User & Narrowest Wedge

**Target user**: ML researcher or team lead on a recommendation systems team at a mid-to-large tech company. They run frequent experiments (A/B tests, hyperparameter sweeps) and need to track which preprocessing strategy improved metrics.

**Specific persona**: Someone who currently uses MLflow for experiment tracking, runs Jupyter notebooks for data preprocessing, and struggles to answer "which data variant gave me the best NE?"

**Narrowest wedge**: API-first lineage tracking for recommendation systems only (NE/CTR/coverage metrics). User calls NanoML SDK from notebook:
```python
import nanoml as dv

dv.init(api_key="sk-...")

# Preprocess data → creates commit in data branch
df_cleaned = df.dropna()
dv.commit_data(df_cleaned, message="v1: remove missing values")

# Run hyperparameter sweep
for sample_rate in [0.1, 0.3, 0.5, 0.7]:
    X_sampled = df_cleaned.sample(frac=sample_rate)
    model = train_model(X_sampled)
    dv.log_model(model, params={"sample_rate": sample_rate})

# Get dashboard URL
print(dv.get_dashboard_url())
# → https://app.nanoml.ai/project/abc123
```

Dashboard shows bipartite graph: data commits (left) × model runs (right), edges showing "trained_on" relationships.

## Constraints

1. **Must integrate with MLflow** (not replace it) — users already have MLflow workflows, NanoML adds data versioning layer
2. **Metadata-only storage** — no dataset uploads to cloud, only content hashes and commit metadata
3. **Local-first with cloud sync** — works offline, syncs metadata to cloud for team collaboration
4. **Vertical launch** — rec systems only (NE, CTR, coverage), not general-purpose
5. **No agent orchestration in v1** — just lineage tracking, not experiment automation

## Premises

1. **Experiment-level data versioning is distinct from lake-level versioning**: v0=raw, v1=deduped, v2=sampled are research iterations, NOT production rollback scenarios (lakeFS's use case)

2. **Bipartite graph abstraction is correct**: Data commits on left, model runs on right, edges show training relationships

3. **MLflow doesn't solve this**: MLflow tracks runs but not input data versions. `mlflow.log_input()` API exists but isn't widely adopted and doesn't provide commit semantics or graph visualization

4. **Vertical focus (rec systems) is the right wedge**: Narrowing to NE/CTR metrics makes value prop concrete ("which preprocessing improved NE?") vs generic ML tooling

5. **Metadata-only storage is viable**: Content-based hashing (SHA256 of DataFrame schema + sample rows) uniquely identifies datasets without uploading TB-scale files

6. **MLflow integration (Option A) beats standalone (Option B)**: Dual instrumentation burden is acceptable because decorator pattern reduces friction:
   ```python
   @dv.track_experiment  # One decorator auto-links data + runs
   def run_experiment(df, sample_rate):
       with mlflow.start_run():
           model = train(df, sample_rate)
           mlflow.log_metric("NE", 0.85)
   ```

## Approaches Considered

### Approach A: MLflow Integration (API-First)

**Summary**: NanoML provides Python SDK for data versioning with Git-like commit semantics. Integrates with MLflow via dual instrumentation — user calls both MLflow APIs (for runs) and NanoML APIs (for data commits). Dashboard queries both NanoML backend (data) and MLflow tracking server (runs) to render bipartite graph.

**Effort**: M (3-4 weeks for MVP: SDK, metadata store, dashboard, MLflow client integration)

**Risk**: Medium — dual instrumentation creates friction (users must remember to call both APIs). Mitigated by decorator pattern for auto-linking.

**Pros**:
- Leverages existing MLflow adoption (users don't abandon current tools)
- Metadata-only storage keeps costs low (no dataset uploads)
- Bipartite graph provides unique value (MLflow doesn't have this view)
- Vertical focus (rec systems) makes value prop concrete

**Cons**:
- Dual instrumentation burden (must call both `mlflow.log_metric()` and `dv.commit_data()`)
- Dashboard must query two backends (NanoML for data, MLflow for runs) — latency risk
- Depends on users exposing MLflow tracking server URL (not all teams do this)

**Reuses**: MLflow's existing experiment tracking, run metadata, metrics storage

### Approach B: Standalone Platform

**Summary**: Build full experiment tracking platform with data versioning, run tracking, and metrics — don't integrate with MLflow, replace it. Users call only NanoML APIs.

**Effort**: XL (6+ months for parity with MLflow features)

**Risk**: High — users already invested in MLflow, switching cost is massive

**Pros**:
- No dual instrumentation (single API surface)
- Full control over data model and UX
- Can innovate faster without MLflow constraints

**Cons**:
- Must rebuild all MLflow features (model registry, artifact storage, UI)
- Users won't switch unless NanoML is 10x better, not 2x
- Longer time to market, higher burn rate

**Reuses**: Nothing (greenfield build)

### Approach C: Quality-First (NanoML SDK as-is + Dashboard)

**Summary**: Keep current NanoML SDK (data quality checks like missing values, duplicates), add lineage dashboard as secondary feature. Don't integrate with MLflow.

**Effort**: S (1-2 weeks to add dashboard to existing SDK)

**Risk**: Low technical risk, HIGH product risk — data quality is table stakes, lineage is the differentiation

**Pros**:
- Fastest to ship (extends existing codebase)
- Data quality checks have clear ROI (find bugs before training)

**Cons**:
- Lineage becomes "nice to have" instead of core value prop
- Doesn't solve the "which preprocessing improved NE?" problem users actually have
- Competes with Great Expectations, Pandera (commoditized space)

**Reuses**: Existing NanoML profiling and validation code

### Approach D: API-First SaaS (Chosen Direction)

**Summary**: Cloud SaaS with Python SDK. Users call `dv.init(api_key)`, commit data versions, log models (via MLflow integration), get dashboard URL. Metadata stored locally (`~/.nanoml/`) and synced to cloud. Dashboard renders bipartite graph by querying NanoML backend (data commits) + MLflow tracking server (model runs).

This is a refinement of Approach A with explicit cloud SaaS architecture.

**Effort**: M (4-5 weeks: SDK, local metadata DB, cloud backend, dashboard, MLflow integration)

**Risk**: Medium — cloud sync adds complexity, API key management, user trust ("what data leaves my machine?")

**Pros**:
- Clear path to monetization (SaaS pricing, API keys)
- Team collaboration (shared dashboard for entire ML team)
- Metadata-only sync addresses data privacy concerns
- Decorator pattern reduces dual instrumentation friction

**Cons**:
- Requires cloud infrastructure (backend API, dashboard hosting, auth)
- Users must trust NanoML with metadata (even though not full datasets)
- MLflow tracking server URL exposure still a dependency

**Reuses**: MLflow client libraries for querying runs, existing NanoML SDK for metadata hashing

## Recommended Approach

**Approach A/D (MLflow Integration with API-First SaaS)** because:

1. **Leverages existing behavior**: Users already use MLflow, NanoML adds data versioning layer without forcing migration
2. **Metadata-only storage is the right tradeoff**: Content hashing uniquely identifies datasets without uploading TB-scale files, addresses privacy concerns
3. **Bipartite graph is unique value**: MLflow doesn't provide this view, solves the "which preprocessing improved NE?" problem directly
4. **Vertical focus de-risks**: Rec systems (NE/CTR) is a concrete wedge vs "ML tooling for everyone"
5. **Decorator pattern mitigates dual instrumentation**: Reduces from 3 API calls to 1 decorator, acceptable friction

**Key insight from user's final clarification**: NanoML versions **experiment preprocessing variants** (v0=raw, v1=deduped, v2=sampled), NOT production data lake rollback (lakeFS's use case). This is the gap between lakeFS (infrastructure-level) and NanoML (research-level).

## Open Questions

1. **Dual instrumentation adoption**: Will users actually adopt the decorator pattern, or will they find it too magical? Should we offer explicit API alongside decorator for power users?

2. **MLflow tracking server access**: What % of users have MLflow tracking server URL exposed? Do we need to support local-only mode (read from `mlflow.db` directly)?

3. **Databricks Unity Catalog overlap**: Databricks has lineage tracking in Unity Catalog. Do we compete or integrate? Is there a gap for non-Databricks users?

4. **Agent orchestration**: User originally wanted agent-driven experiment automation. Is lineage tracking alone valuable, or does it need orchestration to justify adoption?

5. **Pricing model**: Per-user SaaS (like GitHub)? Per-team (like Datadog)? Usage-based (# of commits/runs)?

## Success Criteria

1. **Activation**: User installs SDK, commits first data version, sees it in dashboard within 5 minutes
2. **Value realization**: User runs experiment sweep (3+ data variants × 3+ hyperparams), dashboard shows which (data, hyperparam) combo produced best NE
3. **Team adoption**: 3+ researchers on same team using shared NanoML project (collaboration signal)
4. **Revenue**: $50k-200k ARR per ML team (based on user's stated value — 50% research time savings)
5. **Retention**: User runs `dv.commit_data()` at least once per week for 4 consecutive weeks (habitual usage)

## Distribution Plan

**How users get it**:
- `pip install nanoml` (Python SDK via PyPI)
- `dv.init(api_key="sk-...")` (API key from https://app.nanoml.ai)
- Dashboard hosted at https://app.nanoml.ai/{project_id}

**CI/CD pipeline**:
- GitHub Actions: run tests on PR, publish to PyPI on tag push
- Backend: Vercel or Render for dashboard hosting
- Database: Supabase or Postgres for metadata storage

**Existing deployment infrastructure**: None yet (greenfield SaaS). Current NanoML is SDK-only (no backend).

## Dependencies

1. **MLflow client integration**: Must query MLflow tracking server API to fetch runs for bipartite graph
2. **Content-based hashing**: Need reliable DataFrame hashing (schema + sample rows) for metadata-only storage
3. **Cloud backend**: API for metadata sync, auth (API keys), team management
4. **Dashboard implementation**: React + D3.js for bipartite graph visualization
5. **Local metadata store**: SQLite DB at `~/.nanoml/metadata.db` for offline-first operation

**Blockers**:
- None critical. MLflow is open source (no licensing blocker). Cloud infrastructure is commodity (Vercel/Render/Supabase).
