## Problem Statement

ML teams waste 20-30% of GPU training budget on duplicate experiments and preventable failures. Engineers running 20+ experiments lose track of what they already tried and accidentally rerun similar configurations, wasting thousands of dollars on redundant compute.

The real cost isn't $/hour—it's the massive share of training minutes spent on runs that never should have happened: duplicated experiments (20-30%), broken pipelines (30-40%), data corruption (5-15%).

**Core insight:** Every ML platform today does PASSIVE tracking (MLflow, W&B log what happened). No one does ACTIVE blocking (prevent wasteful runs before they start). NanoML sits between data pipelines and training systems as a pre-execution gate, answering: "Should this GPU run exist at all?"

## Premises

### Validated Premises (from v1)

1. **Duplicate experiments are 20-30% of GPU waste**
   - Confirmed by customer conversations
   - 2nd biggest source (after failed runs 30-40%)

2. **ML engineers frequently run exact duplicates by accident**
   - Engineers running 20+ experiments forget what they tried 2 weeks ago
   - No global memory of "what has already been run"

3. **Dataset hashing is cheap enough via sampling**
   - Sample 0.1% of 500GB dataset (500MB)
   - Complete fingerprinting in <30 seconds
   - Sufficient for duplicate detection

4. **This is a product, not a feature**
   - 20-30% savings + compounding value (irreplaceable database) OR catastrophic loss prevention (one $40K mistake) is sufficient wedge
   - Need either (1) database becomes irreplaceable or (2) prevents one disaster

5. **"Experiment governance" vs "experiment tracking" is the moat**
   - MLflow/W&B are passive trackers
   - NanoML is active pre-execution gate
   - Enforces policies, not just logs

### Revised Premises (CEO review challenges)

6. **REVISED: Near-duplicates are where compounding value lives**
   - Original: "Exact duplicate detection is sufficient wedge"
   - **CEO Challenge**: Exact duplicates are rare (engineers rarely run IDENTICAL configs). Near-duplicates (95%+ similar) are where the waste happens.
   - **Revised**: Near-duplicate detection (95%+ similarity) is REQUIRED for v1.0, not v2.0
   - **Evidence needed**: First user reports "NanoML warned me about similar experiment, saved me from re-running"

7. **REVISED: Outcome linkage creates the moat**
   - Original: "Fingerprint database alone is the moat"
   - **CEO Challenge**: Fingerprint database without outcomes = glorified git log. Anyone can copy it.
   - **Revised**: Store experiment outcome (success/failure/metrics) alongside fingerprint. This enables learning: "We tried 95% similar config—it failed with OOM."
   - **Moat**: Outcome data is unique to NanoML users, can't be copied by MLflow

8. **REVISED: Cost estimation creates immediate ROI clarity**
   - Original: "20-30% savings is abstract until procurement"
   - **CEO Challenge**: Startups won't pay $1-3K/month for abstract savings without proof
   - **Revised**: Show "estimated $4200" for each experiment. This makes ROI visible in month 1.
   - **Activation metric**: User configures GPU pricing on first run

9. **REVISED: 30-second fingerprinting is too slow**
   - Original: "<30 seconds is acceptable"
   - **CEO Challenge**: Engineers won't wait 30 seconds. git status is <1s, docker build is <5s.
   - **Revised**: Add fast path: cached fingerprints (<1s), cloud metadata (<5s), sampling fallback (30s worst case)
   - **Target**: 90% of checks complete in <5 seconds

## Approaches Considered

### Approach A: Lightweight CLI Gate (v1 - ORIGINAL)

**Summary:** CLI tool that users run before launching training jobs. Queries a central fingerprint database, blocks exact duplicates only.

**Effort:** S (1 engineer, 6-8 weeks)
**Risk:** Low (simple architecture)
**Success Probability:** 30%

**Cons (CEO review):**
- Exact duplicates only = one-time value, not compounding
- No cost estimation = ROI not visible until month 2-3
- No team sync = personal tool, not team tool
- 30s fingerprinting = too slow

### Approach A+ (v2 - EXPANDED) ⭐ **RECOMMENDED**

**Summary:** CLI tool with near-duplicate detection (95%+ similar), outcome linkage, cost estimation, and optional team sync. Fast path for fingerprinting (<5s typical).

**Effort:** M (1 engineer, 10 weeks)
**Risk:** Low-Medium (adds 4 features to simple architecture)
**Success Probability:** 70%

**Pros:**
- Near-duplicates create compounding value immediately ("we tried 95% similar—it failed")
- Cost estimation makes ROI visible in week 1
- Outcome linkage enables learning (unique moat vs MLflow)
- Optional team sync enables team adoption without complex backend
- Fast path makes CLI tolerable (<5s typical)

**Cons:**
- 2 extra weeks to ship (10 weeks vs 8 weeks)
- Slightly more complex (4 additional features)
- Near-duplicate threshold needs tuning (95%? 98%?)

**Reuses:** Existing data profiling code, dataset hashing logic, adds similarity detection + outcome tracking + cost calculation

## Recommended Approach

**Approach A+ (v2 - EXPANDED): CLI Gate with Near-Duplicates + Cost Estimation**

**Rationale:**

1. **Addresses all 5 critical gaps** from CEO review:
   - Near-duplicates → compounding value
   - Outcome linkage → unique moat
   - Cost estimation → immediate ROI clarity
   - Fast path → tolerable latency
   - Optional team sync → team adoption

2. **Timeline is acceptable**: 10 weeks vs 8 weeks (25% longer, but 2.3x higher success probability)

3. **Low risk**: Building on proven architecture, adds 4 features (not a platform rewrite)

4. **Validates hypothesis**: If engineers DON'T use near-duplicate warnings, we learn that exact duplicates are sufficient

5. **Creates defensible moat**: Outcome data (success/failure/metrics) is unique to NanoML, can't be copied by MLflow

## Implementation Plan (10 Weeks)

### Week 1-2: Core CLI + Exact Duplicate Detection
- CLI framework (Click/Typer)
- Dataset fingerprinting (sampling-based, 30s worst case)
- Local SQLite storage
- `nanoml check` command
- `nanoml history` command

**Milestone**: User can run `nanoml check train.parquet` and get "No duplicates found"

### Week 3-4: Near-Duplicate Detection + Similarity Scoring
- Cosine similarity on config hashes
- Configurable threshold (`--similarity=0.95`)
- Show similar experiments: "95% similar experiment ran 2 weeks ago"
- Tune threshold based on user feedback

**Milestone**: User sees "⚠️ Similar experiment found (97% match) - Experiment #42 from 2024-05-01"

### Week 5-6: Outcome Linkage + Cost Estimation
- Store experiment outcome alongside fingerprint
- `nanoml log-result <experiment_id> --status=success --metric=0.85`
- GPU cost configuration: `nanoml config set-gpu-price 4.76` ($/hour)
- Show estimated cost: "This experiment will cost ~$3200 (7 days × 4x A100 × $4.76/hr)"
- Show outcome when warning: "Similar experiment failed with OOM error"

**Milestone**: User runs `nanoml check` and sees "Estimated cost: $4200. Similar experiment (95% match) failed with OOM. Consider reducing batch size."

### Week 7-8: Fast Path + Performance Optimization
- Cached fingerprints (local FS metadata + hash)
- If path unchanged in last 24h → use cached fingerprint (instant)
- Cloud metadata support (S3 ETag, GCS md5Hash) for <5s checks
- Sampling fallback for worst case (30s)
- Performance testing on real datasets

**Milestone**: 90% of checks complete in <5 seconds

### Week 9: Optional Team Sync
- `nanoml init --team` enables cloud backend
- PostgreSQL hosted backend (Supabase/Render)
- Cloud sync for fingerprints + outcomes
- Team collaboration: Engineer A's experiments visible to Engineer B
- Keep local-only mode as default (no backend required)

**Milestone**: Two users on same team can see each other's experiments

### Week 10: Polish + Documentation
- Error handling (network timeout, invalid dataset path)
- Documentation site with quickstart
- Example workflows (local training, common failure modes)
- Pilot user testing
- Bug fixes

**Milestone**: Pilot user completes full workflow without errors
